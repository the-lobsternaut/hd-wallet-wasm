1. Entropy: The Source of Randomness

Every HD wallet begins with entropy - random data that forms the foundation of your wallet's security. The quality of this randomness is critical: if an attacker can predict your entropy, they can derive all your keys.

Entropy Requirements by Word Count

WASI Entropy Injection

In WebAssembly environments, the WASM module cannot access system entropy directly. HD Wallet WASM uses a WASI bridge pattern where the host application must inject cryptographically secure random bytes before generating mnemonics.

2. BIP-39 Mnemonic Phrases

A mnemonic phrase is a human-readable representation of your entropy. BIP-39 defines how to convert entropy into a sequence of words from a standardized 2048-word list.

The Conversion Process

  1. Generate entropy - 128-256 bits of random data
  2. Compute checksum - Take first entropy_bits / 32 bits of SHA-256(entropy)
  3. Concatenate - Append checksum bits to entropy
  4. Split into 11-bit chunks - Each chunk maps to a wordlist index (0-2047)
  5. Map to words - Look up each index in the BIP-39 wordlist
plaintext 
Entropy (256 bits):
  8f7d3b2a1c4e5f6a7b8c9d0e1f2a3b4c5d6e7f8a9b0c1d2e3f4a5b6c7d8e9f0a

SHA-256 Checksum (first 8 bits):
  a7

Combined (264 bits):
  8f7d3b2a1c4e5f6a7b8c9d0e1f2a3b4c5d6e7f8a9b0c1d2e3f4a5b6c7d8e9f0aa7

Split into 11-bit chunks (24 words):
  [1150, 1885, 1354, ...] -> [medal, village, spray, ...]

Final mnemonic:
  medal village spray acid barrel ... (24 words)

Wordlist Languages

BIP-39 defines wordlists for multiple languages. Each wordlist contains exactly 2048 unique words, carefully chosen to:

  • Be distinguishable by their first 4 characters (for autocomplete)
  • Avoid similar words that could be confused
  • Be common, easy-to-spell words

3. Seed Derivation with PBKDF2

The mnemonic phrase is converted to a 512-bit (64-byte) seed using PBKDF2-HMAC-SHA512. This process allows for an optional passphrase (sometimes called the "25th word") that adds an extra layer of security.

PBKDF2 Parameters

Parameter Value
Algorithm PBKDF2-HMAC-SHA512
Password Mnemonic phrase (UTF-8 NFKD normalized)
Salt "mnemonic" + passphrase (UTF-8 NFKD normalized)
Iterations 2048
Output Length 64 bytes (512 bits)
Passphrase Considerations

The passphrase creates a completely different seed from the same mnemonic. This provides plausible deniability (different wallets for different passphrases) but also means a forgotten passphrase cannot be recovered. There's no way to determine if a passphrase is "correct" - any passphrase produces a valid seed.

4. Master Key Generation (BIP-32)

The 64-byte seed is used to generate the master extended key using HMAC-SHA512. This creates both the master private key and the master chain code.

plaintext 
Input:  64-byte seed
Key:    "Bitcoin seed" (domain separator)

HMAC-SHA512(key, seed) = 64 bytes
  ├── First 32 bytes  -> Master Private Key (256 bits)
  └── Last 32 bytes   -> Master Chain Code (256 bits)

Master Public Key = privateKey * G (elliptic curve point multiplication)

The chain code is crucial for key derivation. It ensures that knowing a child key doesn't reveal information about sibling or parent keys.

Extended Key Serialization (xprv/xpub)

Extended keys can be serialized to a portable format using Base58Check encoding. The format includes metadata about the key's position in the derivation tree.

Bytes Description
4Version bytes (xprv: 0x0488ADE4, xpub: 0x0488B21E)
1Depth (0x00 for master)
4Parent fingerprint (0x00000000 for master)
4Child index (0x00000000 for master)
32Chain code
33Key data (0x00 + private key, or compressed public key)

5. Child Key Derivation

BIP-32 defines two types of derivation: normal (non-hardened) and hardened. The choice between them has important security implications.

Normal Derivation

Index: 0 to 231-1

  • Uses public key + chain code
  • Can derive child public keys from parent public key alone
  • Enables watch-only wallets
  • Risk: If child private key + parent xpub is exposed, parent private key can be computed

Hardened Derivation

Index: 231 to 232-1 (marked with ')

  • Uses private key + chain code
  • Cannot derive child keys from parent public key
  • More secure against key leakage
  • Trade-off: Cannot create watch-only wallets at hardened levels
plaintext 
Normal Derivation (index < 2^31):
  data = public_key || index (big-endian)
  I = HMAC-SHA512(chain_code, data)
  child_key = parse256(I_L) + parent_key (mod n)
  child_chain_code = I_R

Hardened Derivation (index >= 2^31):
  data = 0x00 || private_key || index (big-endian)
  I = HMAC-SHA512(chain_code, data)
  child_key = parse256(I_L) + parent_key (mod n)
  child_chain_code = I_R

6. BIP-44 Multi-Account Hierarchy

BIP-44 defines a standard path structure that organizes keys by purpose, coin type, account, and address. This enables interoperability between different wallet software.

Path Format

m / purpose' / coin_type' / account' / change / address_index
Level Hardened Description
purpose' Yes 44 for BIP-44, 49 for BIP-49 (P2SH-SegWit), 84 for BIP-84 (native SegWit)
coin_type' Yes SLIP-44 registered coin type (0=BTC, 60=ETH, 501=SOL)
account' Yes Account index for user-level separation (0, 1, 2, ...)
change No 0 for external (receiving), 1 for internal (change)
address_index No Sequential index for address generation

Common Derivation Paths

plaintext 
# Bitcoin (Legacy P2PKH)
m/44'/0'/0'/0/0

# Bitcoin (Native SegWit P2WPKH)
m/84'/0'/0'/0/0

# Ethereum
m/44'/60'/0'/0/0

# Solana
m/44'/501'/0'/0'

# Cosmos
m/44'/118'/0'/0/0

# Polkadot
m/44'/354'/0'/0'/0'
Why Different Paths?

Different coins use different paths to ensure addresses never collide across chains. Some chains (like Solana) derive keys differently and may use hardened derivation at more levels. Always check the chain's recommended derivation path.

7. Security Best Practices

Do
  • Generate mnemonics on air-gapped devices
  • Use hardware wallets for large amounts
  • Store backups in multiple secure locations
  • Use strong passphrases for additional security
  • Verify addresses on hardware wallet screens
Don't
  • Enter mnemonics on websites or apps
  • Store mnemonics digitally (photos, notes)
  • Share your seed phrase with anyone
  • Use brain wallets or predictable entropy
  • Reuse addresses across transactions

Extended Public Key (xpub) Exposure

Sharing an xpub allows others to generate all your public keys (and thus addresses) for that branch of the tree. This is useful for watch-only wallets but reveals your complete transaction history for that account.

xpub + Child Private Key = Parent Private Key

For non-hardened derivation paths, if an attacker obtains both a parent xpub and any child private key, they can compute the parent private key (and all sibling keys). This is why the first three levels of BIP-44 paths are hardened.
